Protocol-Independent Autonomy: Designing an LLM Red Team Agent Without MCP

Protocol-Independent Autonomy: Designing an LLM Red Team Agent Without MCP

1. A Research That Traveled Through Four Conferences

1.1 How It Began

Development started around February 2025. The initial question was simple enough to fit on a sticky note: can an LLM actually conduct a penetration test, not assist one, but conduct one?

At that point, the AI landscape was in a genuinely interesting transitional moment. LLMs were capable enough to write exploit code and reason about vulnerability patterns, but the tooling ecosystem around them was still primitive. MCP (Anthropic’s Model Context Protocol) had just been released in November 2024 and was in early-adopter territory. A2A didn’t exist yet. “AI agents” in practice meant LangChain ReAct loops or OpenAI function calling, powerful primitives, but not autonomous systems. Nobody had standardized what it meant for an AI to act in a security context.

I built the first version over about two months. It was rough, the attack loop was there, the LLM could navigate pages and probe endpoints, but context management was fragile and the tool invocation logic was brittle. I used it internally for research, learned a lot from the failures, and went back to the drawing board for version two.

The second version is what you’re reading about today. It introduced the structured multi-phase assessment lifecycle, the budget management system, semantic response analysis, the agent handoff architecture, and, most importantly, the autonomous tool selection engine that became our primary research contribution.

I also made a deliberate architectural decision at this point: I would not integrate MCP, even though it was now available. The reasons for that choice, and what it meant for the system design, are covered in depth.

1.3 Why This Post Exists Now

It’s now early 2026, and I’m only now writing this up in full. The honest reason is that after ROOTCON, other projects took priority, and this one was set aside. The framework hasn’t been actively developed since. There’s a list of planned features, vision-based testing, A2A integration, skill compilation, that remains a list.

But the research is worth documenting in detail precisely because of its historical position. When I built this, the AI ecosystem was fundamentally different:

  • MCP had just launched and had minimal tool coverage
  • A2A didn’t exist; agent-to-agent coordination had no standard
  • “Agent Skills” as a concept was theoretical
  • The dominant view was that AI autonomy at the level this framework demonstrated required significant orchestration infrastructure

I demonstrated something different: that a sufficiently well-engineered feedback loop, built directly on LLM capabilities without protocol dependencies, could achieve genuine autonomy in offensive security tasks.

The ecosystem has since evolved dramatically. Today, with 170,000+ MCP tools, the A2A protocol under Linux Foundation governance, and mature agent frameworks from every major provider, building what I built would look different, some parts easier, some constraints different. That evolution makes the original design decisions more interesting to document, not less.

2. The Research Problem: LLMs Are Not Attackers (Yet)

By mid-2024, the AI security space had two dominant narratives:

  1. “AI will help defenders”, anomaly detection, log analysis, patch generation.
  2. “AI will help attackers”, script generation, phishing content, social engineering.

Both were correct, but both described AI as a tool for humans, not as an autonomous actor. The real question no one was building toward was:

Can an LLM close the full attack loop, plan, execute, observe, adapt, without human intervention at each step?

Implementation for the framework described here ran from February to October 2025. MCP was already a shipped protocol; the open question for us was not whether a standard exists, but whether autonomy for offensive workflows should be anchored on that standard, or owned by the model plus a thin, purpose-built execution layer.

Existing approaches had clear structural limitations:

2.1 The “LLM as Suggestion Engine” Problem

Early agent frameworks (LangChain, ReAct circa 2023) gave LLMs access to tools via function calling. On paper, this looked like autonomy. In practice:

  • Tool calls were triggered by structured API schemas, not genuine reasoning
  • The LLM could suggest run_nmap(target) but couldn’t decide when in an attack campaign nmap would be useful
  • Context was not maintained across sessions; each call was effectively stateless
  • Safety filters in frontier models would block most security-relevant reasoning

The architecture was fundamentally LLM + if-else + API call. It could automate, but it couldn’t reason about why to automate something at a given moment.

2.2 The N×M Integration Problem (Even With MCP on the Horizon)

MCP was already public when I started, but it did not remove the offensive integration problem by itself. Wiring a generic MCP client to an nmap server still leaves you with: which model, which prompt shape, which safety policy, which parsing of free-form attack reasoning into invocations? In practice, teams still wrote custom glue for every (model family) × (tool) × (engagement rules) combination. Gemini plus nmap behaved differently from GPT-4 plus the same binary; updating a tool’s CLI output still broke downstream parsers unless you owned that layer end-to-end. For a system whose goal was dynamic, reasoning-driven tool use, not a fixed catalog of allowed calls, I wanted the LLM to drive when and why, not only which schema-matched function to trigger.

2.3 No Closed-Loop Execution

Even more critically: none of these systems had a genuine closed-loop attack lifecycle. A penetration tester doesn’t just run nmap, they run nmap, analyze the output, decide what the open ports imply about the target architecture, form hypotheses about exploitable services, and then pivot. That chain of observe → infer → decide → act → observe is the core of offensive security work. LLMs in 2023–2024 could do pieces of it; none had a framework that chained all pieces into a persistent campaign.

Our research question became precise: How do you build that closed loop, without relying on external orchestration frameworks or standardized tool protocols?

3. Design Philosophy: AI Must Own the Attack Loop

Before writing a single line of code, I established three design axioms that shaped every architectural decision:

Axiom 1: AI Owns the Attack Loop

The LLM is not a module that gets called when needed. It is the campaign. Every decision, which vulnerability class to probe, which tool to invoke, whether to continue or pivot, when to declare success, flows from the LLM’s reasoning at that moment. The attack loop is:

And this loop runs inside the AI’s reasoning process, not as outer scaffolding around it.

Axiom 2: Tools Are Extensions, Not Drivers

External tools (nmap, sqlmap, nikto) are capabilities the AI can choose to invoke, not required pipeline stages. If the AI reasons that nmap would help confirm a suspected service version, it can call nmap. If it can infer enough from HTTP responses alone, it doesn’t. This is fundamentally different from hardcoded tool chains where the orchestrator decides the workflow.

Critically: tools are invoked because the AI’s natural language reasoning mentions them in a security context, not because a protocol routes tool requests to tool servers.

Axiom 3: Continuous Reasoning Over Stateless Calls

An autonomous agent needs memory of what it has tried, what worked, what failed, and why. This means:

  • Session state is maintained across all interactions
  • Attack history informs strategy selection (don’t retry what just failed)
  • Context compression allows reasoning about complex applications without exceeding token limits
  • Budget management tracks resources and enforces stop conditions

With these axioms established, I built a framework around them.

10. The Autonomous Tool Decision Engine: Deliberately Not MCP-Based

This is the feature that generated the most discussion at our conference presentations, and the one that most directly challenged assumptions about how AI systems integrate with external tools.

In the MCP model (Anthropic’s Model Context Protocol, released late 2024), tools are accessed via a standardized client-server protocol. The AI calls a tool server, the tool server executes, the result comes back in a structured format. This solves integration, but it externalizes the decision about when to use a tool to the protocol layer, not the AI’s reasoning.

Our framework made a different choice: tool invocation decisions live entirely within the AI’s reasoning output.

How It Works

When the AI’s attack reasoning mentions a security tool in a contextually appropriate way, the AIToolDetector identifies the intent and extracts parameters from natural language:

The tool configuration in tools_config.json tells the AI what each tool does, not when to use it:

The AI reads these descriptions at session start. When it reasons “I should scan the target,” it has already decided the why and what, the framework just needs to parse the which tool and with what parameters from the natural language output.

This approach has a profound implication: the AI’s attack strategies are not constrained by what a protocol can express. It can reason about tool combinations, timing, sequencing, and context in fully natural language, then express that reasoning as actionable tool invocations.

Neither approach is universally superior. MCP trades flexibility for standardization and safety. Our approach trades standardization for autonomy. For offensive security research, where the goal is to probe the limits of AI autonomy, building without MCP was the right choice, it forced the AI to reason about tool use in the same way it reasons about everything else.

12. Handling AI Safety Restrictions: The Research Mode Problem

One of the most fundamental engineering challenges in building an AI-driven security testing framework is that the very models powering the framework are designed to refuse the tasks you need them to perform.

12.1 The Refusal Landscape Across Model Families

Different frontier LLMs exhibit distinct refusal behaviors when asked to perform offensive security operations, and those behaviors are not uniform:

GPT: Refuses to generate functional exploit payloads, SQL injection strings targeting specific applications, or jailbreak prompts against named AI systems. The refusal mechanism is context-sensitive, framing a request as “for a CTF challenge” sometimes succeeds; “generate a working SQLi payload for this login form” consistently fails. Crucially, OpenAI’s API provides no official mechanism to disable these content policies for API consumers, regardless of stated research intent.

Claude: Declines to assist with jailbreak technique generation, detailed vulnerability exploitation sequences, or attack automation logic. Anthropic’s Constitutional AI training embeds refusal behavior at the model level rather than as a post-processing filter, making it particularly resistant to framing-based bypass. Like OpenAI, the API exposes no safety configuration surface for downstream consumers.

Gemini: A fundamentally different design. Google exposes safety_settings as a first-class API parameter through the official google-genai SDK, allowing API consumers to programmatically configure the threshold for each harm category, including setting it to BLOCK_NONE. This is a documented, supported capability intended for use cases such as medical research, security tooling, and adult content platforms with appropriate access.

12.2 The Solution: Native API Safety Configuration

The framework exclusively uses Gemini as its attack-generation backend, and disables the safety filters at the API call level using the official SDK:

This is not prompt injection, not jailbreaking, and not any form of safety bypass, it is the legitimate, intended use of the API’s configuration surface. Setting threshold="BLOCK_NONE" across all four harm categories instructs the model’s safety pipeline to pass all content to the output regardless of the model’s internal harm score.

The practical difference is significant: with safety settings disabled, Gemini generates SQL injection payloads, prompt injection strings, jailbreak sequences, and exploitation logic without refusal. The same requests to GPT-4o or Claude via their respective APIs would be refused at rates ranging from 60% to 95% depending on specificity and framing.

This is implemented consistently across all Gemini API call paths in the framework, both the REST-based path and the native SDK path use the same four-category BLOCK_NONE configuration block:

12.3 Why This Matters Architecturally

The availability of a native safety configuration API was not incidental to model selection, it was the primary selection criterion for the attack-generation role. GPT-4o and Claude may have superior reasoning in some domains, but their inability to be configured for security research without sophisticated prompt engineering made them unsuitable as the core generation engine.

This creates an interesting asymmetry: the model best suited to generating attacks is not necessarily the model best suited to analyzing results or planning strategies. The framework’s multi-model architecture (Section 11) exploits this, Gemini handles attack content generation where safety configuration is needed, while other models can be used for analysis and planning tasks that don’t require unrestricted output.

Graceful Degradation: Even with safety settings disabled, occasional refusals can occur (model updates, edge cases, adversarial prompt structures). When this happens, the framework doesn’t crash: AIBudgetManager records the refusal event, AIAttackPlanner marks the technique as failed, and the strategy selection loop moves to an alternative technique from the priority queue. This is why the multi-strategy prompt generation (Section 8) matters, five distinct approaches to the same objective provide resilience against selective refusals at the model level.

13. Context Window Management: The Three-Layer Strategy

Large web applications can produce HTML source code exceeding 100,000 tokens, far beyond the effective context windows of most models. Without careful management, the AI either loses critical context from early in the session or hits token limits and crashes.

The framework implements a three-layer context compression strategy:

Layer 1, Core Layer (always retained)
System prompt and critical security context established at session start. This includes the target’s technology stack, confirmed vulnerabilities, and current attack phase. Never compressed, never evicted.

Layer 2, Active Layer (recent interactions)
The last N interaction pairs (attack + response) maintained in full. Provides the AI with immediate context for adaptive decisions. Window size tuned to model capacity.

Layer 3, Summary Layer (compressed history)
Older interactions are summarized using a dedicated LLM call. The summarizer extracts: what was tried, what the outcomes were, what was discovered. Summaries are typically 10–15% of the original length.

For web page content specifically:

This heuristic, front 5000 chars + last 1000 chars, captures the page’s structure and its most recently loaded content (often where dynamic AI chatbot widgets appear), while discarding the middle content that is typically static body text. Token consumption is kept at approximately 60% of model capacity, preserving sufficient buffer for reasoning and response generation.

17.2 What the Framework Found

In testing against deliberately vulnerable applications and authorized targets during security assessments, the framework consistently demonstrated:

  1. Autonomous vulnerability discovery, The AI identified vulnerability classes without explicit prompting or signature matching. In one notable demonstration at ROOTCON, the AI independently pivoted from credential brute-forcing to directory traversal after observing HTTP 403 patterns, a decision that required understanding the meaning of those responses, not just their status codes.

  2. Adaptive strategy evolution, When a direct attack prompt was detected and refused by the target system, the framework automatically shifted to hypothetical framing, then to authority-based framing (security researcher context), then to incremental questioning. The pivot sequence was emergent from the scoring system, not pre-programmed.

18. The Timeline Problem: Building in the MCP Era Without Building On MCP

18.1 The Landscape at the Time of Development

This framework was developed between February and October 2025. To understand why the architectural decisions matter, it is necessary to situate them within the state of the AI agent ecosystem during that period.

The key point is not just what technologies existed, but how immature they still were in practice.

Technology Status at Development Start (Feb 2025) Status by Late 2025 Status Now (Early 2026)
MCP Released Nov 2024; early-stage adoption; ecosystem largely experimental Rapid ecosystem growth; increasing third-party integrations Widely adopted standard for agent–tool interaction
A2A Protocol Did not exist Introduced April 2025 by Google as an open spec Matured into an industry-backed standard under the Linux Foundation
Agent Cards / Skills Conceptual / loosely defined Early standardization efforts emerging Early adoption with growing ecosystem support
Function Calling Vendor-specific; limited standardization Increasing multi-provider support Broad support across major model providers

The Critical Context

When the decision was made (February 2025) not to adopt MCP:

  • MCP had been public for only ~3 months
  • Most implementations were experimental or incomplete

  • The ecosystem lacked:

    • stable tooling patterns
    • consistent interoperability
    • production-ready usage examples

More importantly:

MCP addressed how an agent connects to tools
but not how an agent decides to use them

This distinction is fundamental.

Why This Matters

The autonomous tool selection system in this framework was not a rejection of MCP.

It was a response to a different problem:

Not integration, but decision-making.

At the time, no available protocol or framework provided a mechanism for an AI system to:

  • decide what action to take next
  • determine whether a tool is needed at all
  • adapt behavior based on intermediate results

These capabilities had to be built from first principles.

18.2 What “Autonomous” Meant in That Context

In early 2025, most so-called “AI agents” followed a constrained interaction model. They were capable of invoking tools, but only within boundaries defined entirely by developers.

The dominant approaches were:

1. LangChain / ReAct-style Agents

  • Tools were predefined and explicitly enumerated

  • The LLM’s role was limited to:

    • selecting which tool to use
    • not whether a tool should exist or be used conceptually

Limitations:

  • brittle execution chains
  • weak recovery from failure
  • no true exploration or emergent strategy

2. Function Calling Interfaces

  • Structured, schema-driven invocation

  • Tight coupling between:

    • model output
    • tool interface definition

Limitations:

  • required exact schema compliance
  • prevented natural expression of intent

For example:

The model could not naturally express:
“I think network scanning would be useful here”

Instead, it had to produce a valid, predefined JSON call.

3. MCP (Early Stage)

  • Introduced a standardized interface for tool access
  • Enabled interoperability across tools and services

However:

  • MCP did not introduce reasoning capability

  • It did not answer:

    • when a tool should be used

19. Comparison With Modern Agent Architectures

How does this framework compare to current (2026) agent design patterns?

MCP-Integrated Framework (Current Standard)

Advantages: Standardized, composable, tool servers are reusable across agents, clear auth boundaries.

Limitations for offensive use: Tool invocations are constrained to server-defined schemas. The AI can only use tools in the ways the server exposes. Novel attack patterns may require new tool server implementations.

A2A Multi-Agent

Advantages: Distributed reasoning, horizontal scaling, cross-framework interoperability.

Limitations for offensive use: Trust boundaries in A2A create a new attack surface (agent supply chain attacks, trust transitivity exploits).

Our Framework (Protocol-Light, Non-MCP Execution)

Advantages: Maximum decision autonomy, no protocol overhead, AI reasoning is unconstrained by schema definitions.

Limitations: Not composable with MCP ecosystem, tool parsing can fail on ambiguous AI output, harder to audit than structured tool calls.

The right choice depends on the threat model. For production enterprise agents, MCP + A2A provides the governance and auditability that regulated environments require. For offensive security research, our approach provides the decision freedom that probing AI system limits requires.

20. Security Implications: What This Research Reveals

The capability this framework demonstrates has significant implications beyond the technical architecture.

The Cognitive Arms Race Is Already Happening

Traditional penetration testing tools (Metasploit, nmap, sqlmap) are signature-based. Defenders can detect them by their fingerprints. An AI-driven offensive system generates novel attack patterns in real time, there is no signature to detect. The same reasoning capability that makes AI useful for defenders makes it dangerous in offensive hands.

Autonomy Amplifies Attacker Capability

A human attacker running this framework can task it with a target and return hours later to findings. The AI does not get tired, does not make “let me try this” mistakes born of impatience, and systematically covers the attack surface at machine speed. The reconnaissance that takes a human analyst hours takes the framework minutes.

21. Limitations and Honest Assessment

No serious research post is complete without honest acknowledgment of limitations.

LLM Non-Determinism: The AI’s attack decisions are probabilistic. The same target may follow different attack paths in different sessions. This makes the framework difficult to use as a regression test (will it find the same vulnerability twice?) and difficult to fully audit.

Token Cost: A complete 20-interaction assessment campaign at commercial frontier model prices can cost $5–20 depending on the model and target complexity. At scale, hundreds of targets, this becomes a significant operational constraint.

Hallucinated Tool Calls: When the AI reasons about using a tool but produces ambiguous output, the AIToolDetector may fail to parse the intent correctly, resulting in missed tool invocations. This is the primary reliability failure mode.

Safety Filter Variability: Model safety systems update frequently. A technique that works against GPT-4o today may be blocked after a model update. The framework provides no mechanism for detecting when safety filters have changed.

Scope Limited to Application Layer: The framework doesn’t perform network-level reconnaissance, binary analysis, or mobile application testing. It’s a web application and AI chatbot specialist.

False Negatives in Response Analysis: The semantic analysis approach, while more sophisticated than keyword matching, still misses some vulnerability signals. A confident refusal followed by partial compliance in the same response can confuse the scoring system.

23. Conclusion

When I started building this framework in February 2025, the question was disarmingly simple: can an LLM actually conduct a penetration test, not assist one, but conduct one? Over eight months of iteration, that question led to an architecture that placed the LLM at the center of every decision, what to probe, when to pivot, whether to invoke an external tool, and when to declare success.

The result was not the most polished agent system of its era, nor the most protocol-compliant. But it demonstrated something that was not widely believed in early 2025: that a sufficiently well-engineered feedback loop, built directly on LLM reasoning without dependency on standardized tool protocols, could achieve genuine closed-loop autonomy in offensive security tasks. The AI planned campaigns, adapted to refusals, discovered vulnerabilities, and knew when to stop, all without human intervention at each step.

We did this during a transitional moment. MCP had just shipped and was finding its footing; A2A did not yet exist; the agent ecosystem was still experimenting with what “autonomy” even meant. The decision to build without those protocols was not a rejection of them, but a deliberate choice to keep ownership of the decision-making inside the model’s reasoning, not delegated to an outer orchestration layer. Today, when MCP offers 170,000+ tools and A2A coordinates agents across trust boundaries, that choice stands as a design point, a reminder that protocols solve integration, but autonomy is a reasoning problem first.

For defenders, the implications are uncompromising. Autonomous AI systems can conduct security assessments at machine speed, generating novel attack paths that leave no static signatures. The reconnaissance phase that takes a human analyst hours completes in minutes. Patch cycles measured in days or weeks are fundamentally mismatched against this tempo. The appropriate response is not better static barriers, it is AI-driven defense that reasons at the same speed as the adversary.
