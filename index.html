程式碼分解

運作原理

這段程式碼中有兩個 for 迴圈：

  1. **外層 for i in range(1, n + 1)**：控制「行數」。
  2. **內層 for j in range(1, n + 1)**：控制每一行的「列數」。

讓我們逐步解釋程式如何運作，假設使用者輸入 n = 3。

步驟分析

  1. 使用者輸入 n = 3。
  2. 程式進入外層迴圈，i 會從 1 開始，依次取值 1, 2, 3。
    • 當 i = 1 時，程式會執行內層迴圈。
    • 當 i = 2 時，再次執行內層迴圈。
    • 當 i = 3 時，最後一次執行內層迴圈。
  3. 內層迴圈 for j in range(1, n + 1)：
    • 對於每個 i 的值，j 從 1 到 3，依次執行程式。
    • 每次內層迴圈執行時，都會計算 i * j，並顯示乘法表格的一部分。

更詳細的運作流程

當 n = 3 時，執行的過程如下：

  1. 當 i = 1 時：

    • 進入內層迴圈，j 從 1 到 3。

  2. 當 i = 2 時：

解釋 format(j, i, (i * j)) 的部分

在 print('{} * {} = {:<4}'.format(j, i, (i * j)), end='')：

  • j 和 i 表示當前要計算的乘法因子。
  • (i * j) 是計算的結果。
  • '{:<4}' 格式化輸出會讓結果左對齊，占 4 個空間，讓輸出看起來整齊。
  • end='' 是為了避免 print() 預設的換行，這樣每次輸出會排在同一行，直到內層迴圈結束後再換行。

詳細解釋

  1. 外層 for 迴圈：for i in range(int(input())):

    • int(input()) 會先讓使用者輸入一個數字，表示有幾筆測試資料。
    • 這個 for 迴圈會依次處理每一筆測試資料。

  2. 輸入測試資料並轉成字串：n = str(input())

    • 讓使用者輸入一個正整數 n，然後將 n 轉成字串型別。將 n 轉成字串的原因是為了方便逐位處理，例如 123 可以轉為字串 "123"，這樣每一位可以被當成一個字符來處理。

  3. 列表生成式：a = [int(j) for j in list(str(n))]

    • 這是整段程式中較難理解的部分。
    • 逐步解釋這段代碼：
      • str(n)：將 n 轉為字串形式（如果它還不是字串）。
      • list(str(n))：將字串 "123" 轉為列表 ['1', '2', '3']，每個字符都是一個單獨的字元。
      • [int(j) for j in list(str(n))]：這是一種列表生成式語法，逐一取出字元 j 並將它轉為整數 int(j)，然後存入新的列表 a 中。
      • 最終，a 會變成一個整數列表，比如對於 "123"，a 會是 [1, 2, 3]。

  4. 計算位數總和並輸出：print("Sum of all digits of " + str(n) + " is " + str(sum(a)))

    • sum(a)：計算列表 a 中所有元素的總和。例如，a = [1, 2, 3]，則 sum(a) 為 1 + 2 + 3 = 6。
    • print("Sum of all digits of " + str(n) + " is " + str(sum(a)))：將結果格式化為字符串並輸出，例如 Sum of all digits of 123 is 6。

  • 不必將EBS Volume 從EC2實例中分離
  • 可以跨不同的可用性區域甚至跨不同的區域複製這些EBS快照

  • 封存快照 Archive

    • 快照移動到封存層
    • 成本最多可以降低75%
    • 恢復歸檔將需要24~72小時

  • 回收站EBS快照 Recycle Bin

    • 如果刪除EBS Snapshots，不是永久刪除，會在Recycle Bin中
    • 可以從意外刪除中恢復
      • 可以設置為一天到一年的保留時間

  • 快速快照恢復 Fast Snapshot Restore (FSR)

    • 強制快照進行完全初始化，以便在第一次使用時沒有延遲
    • 如果快照很大，需要初始化EBS或需要快速從其中實例化出來
    • 需要花費大量資金

EBS Volume Types

  • 6 types
    • gp2/gp3 (SSD)
      • 平衡各種工作負載的價格和性能
      • 為了實現經濟高效的存儲延遲
      • gp2
        • 具有低延遲的經濟高效的存儲
        • 可將其用於系統引導Volume
        • 1GB~16TB不等
        • 有些較小的gp2 Volume 可以爆發到3000 IOPS
        • Volume 的大小和IOPS關聯
        • IOPS、吞吐量連接再一起無法單獨設置
        • 如果增加IOPS、Volume GB數，將獲得3個以上的IOPS，最高到16000 IOPS ⇒ 5334 GB(最大值)
      • gp3
        • 新一代Volume
        • 可獨立設置IOPS和吞吐量
        • 提供了3000 IOPS的基線和每秒125MB的吞吐量
        • 可以將IOPS增加到16000
        • 吞吐量增加到每秒1000MB
    • Provisioned IOPS (PIOPS SSD) Input/Output Operations Per Second

      • 用於需要維持IOPS性能的關鍵業務應用程式

      • 需要大量的IOPS(可能超過16000的應用程式)

      • 存儲性能和一致性非常敏感的數據工作負載

      • io1(4GiB-16GiB(最大可調配))

        • 對於Nitro EC2實例，最大PIOPS約為64000
        • 對於其他類型的實例，最大PIOPS約為32000
        • 可以獨立於存儲大小來增加調配的IOPS

      • io2 Block Express(4GiB-64TiB)

        • Sub-millisecond 等級的延遲
        • 獲得大量的最大IOPS
          • 256000 with an IOPS:GiB ratio of 1000:1
        • 性能非常高的I/O類型的Volume

      • 支持EBS多連接功能

        • 允許將相同的EBS Volume 連接到同一可用性區域中的多個EC2實例
        • 僅適用io1、io2系列的EBS Volume
        • 每個實例都將擁有對高性能Volume 的完全讀寫權限，意味著可以同時讀寫
        • 在clustered Linux Application(Teradata 或你的應用程式必須管理併發寫入操作) 將獲得更高的應用程式可用性
        • 僅在指定可用性區域中可用
        • 不允許將EBS Volume 從一個連接到另一個
          • 多連接的另一個限制是一次最多可以連接16個EC2實例
        • 要使其正常工作，必須使用一個支持集群的文件系統
          • 不同於XFS、EXT4

      • io1/io2 Block Express (SSD)

        • 性能最高的SSD Volume
        • 用於任務關鍵型低延遲和高吞吐量工作負載
    • Hard Disk Drives
      • 不能是Boot Volume
      • 僅適用於以前類型的Volume
      • 125GiB - 16 TiB
      • st1 (HDD)
        • 經吞吐量優化的HDD
        • 適合大數據、數據倉庫、日誌處理
        • 最大吞吐量為每秒500MB/s
        • 最大IOPS 為500
        • 為頻繁訪問的吞吐量密集型工作負載而設計的低成本HDD Volume
      • sc1 (HDD)
        • 成本最低的HDD Volume
        • 設計用於訪問頻率較低的工作負載
        • 用於歸檔數據
          • 不經常訪問的數據
        • 當需要盡可能低的成本時可以使用它
        • 最大吞吐量為每秒250 MiB/s
        • 最大IOPS為250
  • 如何定義EBS Volume ?
    • 考慮 Size |Throughput IOPS (I/O Ops Per Sec)
  • 通用SSD與預配置IOPS SSD(如果需要數據庫)與st1、sc1(如果需要高吞吐量和最低成本)
  • 如果想要獲得超過32000 IOPS，則需要具有io1 or io2的EC2 Nitro

EBS Encryption

  • 當創建並加密EBS Volume 時
    • Volume 中的靜態數據被加密
    • 實例和Volume 之間傳輸的所有數據都被加密
    • 所有快照都將被加密
    • 從快照創建的所有Volume 都將被加密
  • 所有加密和解密機制都透明的處理
    • 一切都由EC2和EBS在後台處理
  • 對延遲的影響非常小
  • 利用了KMS(AES-256)的密鑰
  • Copy 未加密的快照時，就啟用了加密
  • 如何加密未加密的EBS Volume
    • 創建該Volume 的EBS快照
    • 複製功能加密EBS快照
    • 從快照創建一個新的EBS Volume
    • 該Volume 也將被加密
    • 將加密的 Volume 連接到原始實例

EFS 彈性文件系統 Elastic File System Service

  • 託管NFS (Network file system)

  • 網路文件系統

  • 可以掛載到許多EC2實例上

    • EC2可以位於不同的可用性區域

  • 高度可用，可自動擴展，昂貴

  • 按使用量付費

    • 用於EFS中的每GB數據

  • 不必提前配置容量

  • 內容管理、web 服務、數據共享、Wordpress

  • 內部使用NFSv4.1 協議

  • 為了控制EFS的訪問，需要設置安全群組

  • 只兼容基於Linux的AMI，而不是Windows

  • 使用KMS在EFS驅動器中啟用靜態加密

  • 它是Linux上的標準文件系統，因此使用POSIX系統，具有標準文件API

  • EFS規模

    • 可以獲得數千個併發NFS客戶端和10GB+/s吞吐量
    • 自動擴展到PB級的網路文件系統

  • 創建EFS網路文件系統時設置性能模式

    • 通用(默認)
      • 延遲敏感的用例(WEB server, CMS..)
      • 最大化吞吐量
        • 最大的I/O
        • 高延遲的網路文件系統
        • 更高的吞吐量
        • 高度並行
        • 大數據應用程式或媒體處理需求

  • 吞吐量模式

    • Bursting 爆裂
      • 一個TB，每秒50MiB 加上每秒100MiB
    • Provisioned 資源調配
      • 無論存儲大小如何都要設置吞吐量
      • 隨存儲容量的增加，前一個系統的吞吐量也在增長
      • 通過配置可以為1TB的存儲容量提供每秒1GB的吞吐量
    • Elsatic
      • 根據工作負載自動調整吞吐量
      • 適合不可預測的工作負載

  • 存儲類型

    • 生命週期管理功能

    • Standard : 允許在一定天數後將文件移動到不同的存儲層

      • 頻繁訪問的文件標準層

    • Infrequent access (EFS-IA):

      • 可以檢索文件
      • 存儲文件的成本低
      • 不頻繁訪問的EFS-IA層

    • Archive 歸檔存儲層:

      • 針對很少訪問的數據
      • 數據存儲在該層上的成本低，50% cheaper
      • 要在存儲層之間自動移動文件，可以實施生命週期策略
      • 允許定義文件在多少天後應移動到哪個層

  • 可用性和持久性方面

    • Standard
      • 當具有多個可用區設置
      • EFS跨多個可用區
      • 生產工作負載非常有用
    • One Zone
      • 只想進行開發並希望有更便宜的選擇
      • 可以選擇一個區域，將提供一個可用性區域，仍然可以在其中進行備份，並與IA類型的存儲層兼容(EFS One Zone-IA)

  • 通過使用正確的EFS存儲類型，可以節省高達90%的成本

  • EBS volumes

    • 一次連接到一個實例
      • 除了使用io1和io2類型volumes的多連接功能邊緣情況外
    • 在AZ級別鎖定
    • gp2
      • 如果磁盤大小增加，IO將增加
    • gp3 & io1
      • 獨立於磁盤大小增加IO
    • 要跨AZ遷移EBS，需要拍快照
      • 將快照恢復到另一個AZ
      • EBS volumes備份，將使用大量IO，不應該在應用程式處理大量流量時運行他們，因為可能影響性能
    • 如果EC2被終止，實例的路由EBS volumes將默認被終止
      • 可以禁用該行為

  • EFS volumes

    • 網路文件系統
    • 目標是將其連接到可用性區域中的數百個實例
    • 對於一個EFS，可以在不同的AZ中擁有不同的掛載目標
    • 多個實例可以一起共享該文件系統
    • 只適用Linux(POSIX)
    • EFS價位高於EBS，但可以利用存儲層節省成本
    • 對於實例存儲，它是物理連接到EC2實例的，因此如果失去了EC2實例，你也會失去存儲

What is prototype pollution?

它允許攻擊者控制原本無法訪問的對象的屬性。如果應用程式隨後以不安全的方式處理攻擊者控制的屬性，則可能會與其他漏洞連結。在用戶端 JavaScript 中，這通常會導致 DOM XSS，而伺服器端原型污染甚至會導致RCE。

攻擊者可能會使用包含有害值的屬性污染原型，這些屬性隨後可能被應用程式以危險的方式使用。

任何原型物件都可能受到污染，但這種情況最常發生在內置的全域 Object.prototype 中。

    • 這樣的注入可能會影響到應用程式或庫的行為，因為它們可能會使用這些屬性來執行特定的功能或操作

重點 2: 緊湊耦合 vs. 鬆散耦合
符號: 🔗 vs. 🔗🔄

  • 定義:
    • 緊湊耦合: 元件之間耦合緊密，一個元件的故障可能影響整個系統。
    • 鬆散耦合: 元件之間耦合較鬆散，一個元件的故障不會影響整個系統。
  • 舉例流程:
    • 緊湊耦合: 應用程式 A 直接傳送訊息給應用程式 B -> B 故障 -> A 開始發生錯誤
    • 鬆散耦合: 訂單放置到佇列 -> 咖啡師取得訂單 -> 製作飲料 -> 移除已完成訂單

  • 舉例流程: 訂單放置到佇列 -> 咖啡師取得訂單 -> 製作飲料 -> 移除已完成訂單

重點 4: AWS 服務: Amazon Simple Notification Service (SNS)

符號: ⬛ vs. 🔲

    • 微型服務: 元件之間耦合較鬆散，一個元件的故障不會影響整個應用程式。

  • 舉例流程:

    • 單體式應用程式: 元件耦合緊密 -> 故障可能導致整個應用程式失效
      • 具備緊耦合元件的應用程式，這些元件可能包括資料庫、伺服器、使用者介面、商業邏輯等
    • 微型服務: 元件彼此鬆散耦合 -> 故障不會影響整個應用程式
      • 服務和元件滿足不同的功能
      • 有兩種服務可促成應用程式的整合：Amazon Simple Notification Service (Amazon SNS) 和 Amazon Simple Queue Service (Amazon SQS)
